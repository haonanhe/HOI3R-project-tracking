User flagged two v2 symptoms: (1) scene point prediction looks poor in camera-view panel, both frozen and unfrozen; (2) hand/obj regressed from v1. The Phase-1 diagnostic found the root cause: v2's gt_pts3d was built in DexYCB-world-frame (via real cam2world with a 1.64 m translation and 156° rotation), but STream3R's point_loss supervises predictions in first-view-camera-frame. The 156° rotational mismatch made the regression target unreachable, so the conf_loss optimiser collapsed pred world_points to a near-constant 0.25 m plane (v2 A: 55.7% valid pixels, z range [0.05, 0.47] m).

Fix (commit c90bd52, one-line change): pass np.eye(4) to depthmap_to_absolute_camera_coordinates when building the GT pointmap — for a single static camera this is exactly X_cam, matching what the model predicts.

Fix verified. v3 world_points on frame 67:

Residual bug — loss_pts3d_grad / loss_depth_grad are still constants

Even after the coordinate fix, the gradient-of-pointmap loss terms report identical values every step:

  • loss_pts3d_grad: 0.0130 at step 1 and 0.0130 at step 200 (v3 A and v3 B)
  • loss_depth_grad: 0.0389 or 0.0390 throughout both runs

Original hypothesis: these were stuck because pred collapsed to a plane (grad(const) = 0 ⇒ grad(pred − gt) = −grad(gt) = constant). That explanation is now refuted: v3 pred has real structure (z in [0.24, 4.6] m) and the gradient loss values still don't budge.

Remaining hypothesis: gradient_loss_multi_scale averages across 4 scales on a downscaled pointmap; the PIL NEAREST resize we use for depth may produce block-constant depth at the coarsest scales, making the gradient image loss mostly insensitive to pred changes. Worth a follow-up fix but not blocking — the conf components of both pts3d_loss and depth_loss do move (see curves above), so the scene head still gets useful signal.

Interpretation & policy update

Scene-loss pipeline is now correct. With the coordinate fix, point_loss supervises pred world_points in the frame the model actually predicts in. Scene points recover fully on both frozen and unfrozen runs.
Freezing policy — update. v3 gives the clearest picture so far:
  • Frozen A: hand 33.4 / obj 20.8 mm
  • Unfrozen B: hand 32.8 / obj 12.6 mm — best object L2 across every run to date
The "freezing wins" verdict from v1/v2 is now reversed for object L2. Unfreezing still slightly loses on hand (~1 mm, noise) but wins substantially on object. This inverts the smoke-stage policy in docs/PROJECT_PLAN.md: with correct supervision, unfreezing helps even on 5 frames. Plan update landing separately.
Caveats:
  1. Still a 5-frame smoke. Full-sequence Phase-3 re-run is where we should cement the policy.
  2. Residual static-grad bug (above) means we're still running with only 3 of 5 scene components effectively active. Expected to strengthen supervision further when fixed.
  3. Hand L2 jumped v1→v3 (22.5 → 33 mm). This is not regression per se — the backbone is now shared fairly between DCF hand trajectory and scene heads. v1 had the backbone fully owned by DCF (scene loss was a weak signal). A full-dataset run should recover hand performance as the backbone has more capacity to serve both.

Hand-shape collapse under unfrozen scene heads — new diagnostic finding

Follow-up observation (post-v3 publish): while v3 object L2 is the best across every run (12.6 mm unfrozen), the hand predicted by the unfrozen runs collapses to a near-planar configuration. Pred-hand L2 is similar to frozen runs, but the pred-hand geometry has lost its 3D rank.

Evidence — singular value decomposition of pred_hand (128 points, frame 67)

After centering the 256 predicted hand points, the three SVD singular values reveal the spatial rank. Ratio s3/s1 ≈ 1 means isotropic 3D blob; s3/s1 → 0 means degenerate line/plane. GT hand has s3/s1 = 0.40 (healthy 3D).

Pattern is stark and version-independent: every frozen run keeps s3/s1 ≥ 0.20; every unfrozen run collapses to s3/s1 ≤ 0.07. The coordinate fix (v3) doesn't touch this — it's purely a function of whether scene heads are trained.

Root cause

The shared aggregator.global_blocks feeds both the scene heads (point_head, depth_head) and the DCF branch (via aggregator_extra_output_list[-1] → per-point hand tokens). Two dynamics:

  1. Scene loss prefers low-rank features. With a static camera + 5 frames, the "best" scene representation is a single consistent pointmap — smooth, low-rank, per-pixel-coherent.
  2. Unfrozen heads amplify the scene pull. With heads frozen, scene gradient flows scene_loss → head(x).jacobian → features → global_blocks — constrained by the fixed head. Unfreezing adds a second path: scene_loss → head.weights + scene_loss → features, and the head can adapt to make current features produce better predictions. The combined system drifts into scene-loss-optimal feature subspaces more aggressively.
  3. DCF hand tokens lose per-point identity. The 256 hand-point tokens are sliced from the same aggregator output. When aggregator features are low-rank, the 256 tokens also become low-rank → DCF MLP's 3D output is low-rank → hand is a plane/line.
  4. trajectory_3d_loss = F.l1_loss doesn't penalise rank loss. A line through the hand's centroid has nearly the same mean-L1 as a proper cluster when the cluster is ~30 mm wide.

Mitigations (ordered)

  1. Keep scene heads frozen for smoke stages. Already the default. Smoke-stage freezing policy in docs/PROJECT_PLAN.md retained.
  2. Switch hand loss to Chamfer or add a rank regulariser. Bidirectional Chamfer forces each pred point to match some GT point and vice versa — a line can't achieve low Chamfer because GT has 3D spread.
  3. Log SVD(pred_hand) per step. Cheap addition; catches rank collapse live instead of only post-hoc.
  4. Lower --lr-scene-heads (e.g. 1e-6) or add EMA-to-frozen. Reduces how aggressively the scene-head training pulls backbone features.

What this means for the v3 headline

The earlier v3 claim "unfrozen wins on object L2 (12.6 mm)" remains true on the reported metric, but the hand predictions for v3 B are geometrically degenerate despite a reasonable 32.8 mm L2. Before concluding "unfrozen is better", we need either a Chamfer hand loss that penalises rank collapse, or a visual/SVD check in every run. The freezing policy in docs/PROJECT_PLAN.md has been updated to reflect this, with required pre-conditions listed before unfreezing in Phase 3.
