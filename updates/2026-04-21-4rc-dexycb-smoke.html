What is 4RC?

4RC (Four-Head Recurrent Correspondence) is a video scene reconstruction backbone that extends the DUSt3R / STream3R family with four specialized heads operating on DINOv2 ViT-G tokens:
  • depth head — per-pixel 3D point cloud in each frame's camera space
  • confidence head — per-pixel aleatoric uncertainty for the depth prediction
  • track head — per-pixel 3D correspondence to a canonical reference frame
  • track_multi head — multi-scale / long-range tracking correspondence
Additionally the model outputs a camera pose head (extrinsic W2C + intrinsic K) per frame. 4RC processes all T frames in a single causal pass, maintaining a recurrent state. At 1.52 B parameters it is substantially larger than STream3R (which we previously used as our backbone) and covers dynamic-scene tracking, making it the natural candidate for HOI scenes with hand and object motion.

Why migrate from STream3R to 4RC? STream3R gives excellent static-scene point clouds but has no explicit dynamic-correspondence head — critical for disentangling hand/object motion. 4RC's track outputs give us a differentiable bridge to the NLF foreground head: each hand/object vertex can query its 3D correspondence across frames without requiring any parametric fitting at training time.

What this smoke proves

Phase A bring-up goal: verify that the 4RC backbone loads, runs end-to-end in our environment, and produces a plausible point cloud on a real DexYCB clip.

This is not a trained result. 4RC pretrained weights are evaluated out-of-the-box on a DexYCB video. The scene content (indoor hand-object interaction) differs from 4RC's training distribution. We are checking for:
  1. Model loads without error (HF snapshot, config fabrication workaround)
  2. Forward pass runs on H100, memory stays within budget (<12 GiB)
  3. Output tensors have expected shapes (9 keys, all assertions pass)
  4. Point cloud is geometrically plausible (depth range ~ 0.4–2.8 m, not NaN/Inf)
  5. Extrinsic convention is verified as W2C (matches DUSt3R family)
All five criteria pass. The backbone is ready for fine-tuning integration.
Do not interpret the point cloud quality as a model metric. 4RC was not trained on DexYCB; visual artifacts (depth errors on hands/objects, blurry boundaries) are expected and not indicative of the final HOI3R result.

Output schema — 9 keys per frame

Each element of predictions['views'] (list of T=16 dicts) contains:

KeyShape (B=1, H=W=504)Meaning
pts[1, 504, 504, 3]3-D point for every pixel in this frame's camera space (X, Y, Z in metres)
conf[1, 504, 504]Per-pixel confidence logit (apply sigmoid for ∈ [0,1])
track[1, 504, 504, 3]3-D correspondence to canonical reference frame (single-scale track)
conf_track[1, 504, 504]Per-pixel confidence for track
track_multi[1, 1, 504, 504, 3]Multi-scale / long-range 3-D tracking correspondence
conf_track_multi[1, 1, 504, 504]Per-pixel confidence for track_multi
track_query_idx[1]Index of the canonical reference frame used for track heads
extrinsic[4, 4]W2C pose matrix (world-to-camera, SE3). Note: upstream _postprocess_output drops the batch dim — harmless at B=1.
intrinsic[3, 3]Camera intrinsic matrix K. Same batch-dim drop note applies.

Key findings

1. Backbone runs end-to-end on H100. Single forward pass for T=16 frames: 9.44 s, peak VRAM 10.68 GiB. Well within the 40 GiB H100 budget. Shapes verified via assert for all 9 output keys.
2. Extrinsic convention confirmed: W2C. Verified via z-sign check: transforming world-origin-at-scene-centroid by extrinsic gives positive Z (scene in front of camera), matching the DUSt3R / STream3R W2C convention. The extrinsic matrices across frames are near-identity (sub-degree rotation, sub-mm translation) — consistent with a static or very slowly moving scene across 16 frames at 4 fps.
3. Depth range is geometrically plausible. Z statistics: min=0.37 m, p5=0.49 m, median=1.59 m, p95=2.50 m, max=2.86 m. This matches a table-top hand-object scene at roughly arm's length from the camera.
4. Confidence distribution is bimodal. After sigmoid: p5 = 0.73, median = 0.73, p95 = 0.98. The low end (p5 = p50 = 0.73) suggests the model assigns most pixels moderate confidence, with a high-confidence tail at p95 = 0.98 corresponding to well-textured regions. The saturated-low mode may reflect domain shift (4RC trained on diverse video, not specifically hand-held-object scenes).
5. Load-path gotcha: no config.json in upstream HF repo. The Luo-Yihang/4RC HuggingFace repo ships only weights + processor config, not a model config. from_pretrained("Luo-Yihang/4RC") fails. Workaround: fabricate a minimal config.json into the HF snapshot directory (3634b324...) before calling from_pretrained(SNAP_DIR). This is documented in docs/third_party_4rc_note.md (commit 6a019bc).
6. Upstream bug in _postprocess_output: batch dim dropped for extrinsic / intrinsic. At B=1 this is harmless (shape [4,4] and [3,3] instead of [1,4,4] / [1,3,3]). Must be accounted for if we ever run B>1 inference or integrate into a batched training loop. A reshape guard will be added during Phase B fine-tuning integration.
