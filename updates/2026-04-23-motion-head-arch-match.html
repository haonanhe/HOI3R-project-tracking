Why this refactor matters

A prior audit (memory project_motion_init_mismatch.md, ckpt 8c72a09) found that HandMotion / ObjMotion structurally diverged from 4RC's MotionDecoder:

  • AdaLN style differs. 4RC = DiT-style 6·D (shift/scale/gate × {MSA, MLP}); HOI3R used post-attention LN + scale + shift (2·D).
  • No pre-attention LayerNorm. 4RC: x + attn(norm1(x)). HOI3R: q + self_attn(q, q, q).
  • No LayerScale. 4RC uses ls1.gamma / ls2.gamma for residual gating.
  • Cross-attn K/V share one context (no pre-norm). 4RC applies normy(y) separately.

Consequence: only 56 of ~166 per-head params transferred through load_4rc_motion_init_into, so --motion-init-from-4rc was de facto a random init with some MLP/QKV seeding — not a real warm-start.

New structure — byte-identical to 4RC

Each HOI3R motion-head layer is now a (cross_block, self_block) pair, where both blocks are instantiated directly from the 4RC layer library (third_party/4RC/arc/models/arc/layers/block.py):

  • cross_block → 4RC CrossBlock — norm1, normy, CrossAttention (split toq/tok/tov, qk_norm = True), ls1, norm2, Mlp, ls2 (LayerScale init = 0.01).
  • self_block → 4RC AdaLNBlock — norm1, fused qkv, qk_norm, proj, norm2, Mlp, and the DiT-style 6·D adaLN_modulation sidecar (SiLU → Linear(D, 6D), zero-init). No LayerScale (AdaLNBlock deletes it).

Forward ordering within each layer follows 4RC motiondecoder.py:146-177: cross then self. Module path blocks.{i}.self_block.X matches the 4RC checkpoint path motion_decoder.self_blocks.{i}.X after a single prefix rewrite.

The only intentional differences from 4RC scene MotionDecoder:

  • Token counts per head: hand = 2·N_H = 256 slots, object = N_O = 256 — vs. 4RC's 1 + P≈484 patches.
  • RoPE disabled (HOI3R motion queries are 1-D token streams, not a 2-D image grid).
  • Colocated output projections delta_proj (Linear(1536, 3)) and uncert.proj (Linear(1536, 1)) — 4RC factors these into a separate track head.

Open follow-ups

  • STAGE2_FULL recipe should default to --motion-init-from-4rc now that the flag is a real warm-start. Pre-refactor it only seeded ~34 % of params so the flag was mostly decorative; post-refactor it seeds every norm / LayerScale / QKV / AdaLN tensor, which should shorten the cold-start plateau in the motion heads. Worth a smoke A/B on the same subject-01 clip (arm-A random, arm-B 4RC-init) with the new architecture to confirm the warm-start signal is real before committing it to the default recipe.
  • 4RC weights do NOT semantically match hand/object motion. 4RC MotionDecoder was trained on generic scene-point tracking; HOI3R adapts the same modules to hand-joint and object-surface motion. Warm-starting only the norm/attn projections (not the delta/sigma output heads) is intentional — the last linear layer must reset for the new geometry. Expect early-training convergence to look similar in the first ~100 steps regardless of init, then diverge.
  • Arm-B re-run on new architecture. The closed arm-A/arm-B pair at 5cc5faf measured init-impact on the old architecture where only 34 % of params could transfer. A clean re-run on the new architecture will give the first real "4RC-init vs random-init on structurally matched heads" comparison.
  • Optional: expose 4RC's init_values as a config knob. Currently hard-coded at 0.01 (matching 4RC). If arm-A (random init, new arch) diverges, worth sweeping init_values ∈ {1e-5, 1e-4, 1e-2}.
