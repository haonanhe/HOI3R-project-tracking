2026-04-15 · Why Frozen > Unfrozen on Hand Rank — Root Cause

ROOT CAUSE IDENTIFIED: gradient magnitude balance Branch: feat/dcf Commits: c90bd52, 44446b7

Across v1, v2, v3, v4 every unfrozen run (except those with Chamfer) collapses pred_hand toward a plane/line (SVD s3/s1 ≤ 0.10) while every frozen run stays 3D (s3/s1 ≥ 0.20, GT=0.40). This page traces the mechanism.

Final answer: the issue is gradient-magnitude balance on the shared aggregator.global_blocks. When scene heads are unfrozen, scene loss's effective gradient on backbone features grows larger (the head adapts to absorb pred errors, unlocking more feature directions the loss can push on). That larger gradient pulls backbone features toward a scene-loss-optimal low-rank configuration — smooth per-pixel, suited for a single static 5-frame scene — and the DCF hand head, reading from the same backbone, inherits the low-rank. The fix is NOT "keep heads frozen" — it's "keep loss_weight_scene · backbone_gradient commensurate with DCF's gradient". In v5 we dropped loss_weight_scene 10× (1.0 → 0.1); hand SVD recovered from 0.035 (line) to 0.373 (matches GT 0.40), and all three metrics — hand L2 20.6 mm, obj L2 14.1 mm, SVD 0.373 — beat every prior run.

Phase 2 — cheap diagnostics

H5 refutation — no BN stats to drift

grep BatchNorm across stream3r/models/components/heads/ returned nothing. Only LayerNorm appears (3× in camera_head.py, 1× in dpt_head.py). LayerNorm doesn't accumulate running statistics — per-sample normalisation is fixed; only the learnable affine γ/β adapts when unfrozen. That's a special case of H6 (widening effective Jacobian), not an independent mechanism.

H8 refutation — collapse is hand-natural, not camera-Z

Diagnosis

Mechanism

The aggregator.global_blocks features are shared between the scene heads and the DCF hand head (via aggregator_extra_output_list[-1]). Both heads exert gradient pressure on the same features:

  • Scene loss sees ~268k pixels/view × 4 views × 5 components = ~5.3M gradient contributions per step, optimising for a static-scene-consistent low-rank pointmap (the per-pixel GT has strong spatial correlation on this small sample).
  • DCF hand loss sees 256 points/view × 1 view × 1 component = ~256 gradient contributions per step, optimising for per-vertex 3D discrimination.

The ratio 5.3M : 256 ≈ 20,000 already favours scene. When scene heads are unfrozen, the head's weights absorb some of the scene loss — freeing the backbone to be driven further in the low-rank direction — and the effective ratio grows. DCF hand features lose their per-point variance and the MLP head's 3-D output collapses in the already-thinnest direction (palm normal).

Dropping loss_weight_scene to 0.1 multiplies the numerator by 0.1, restoring the balance. The loss is still nonzero (scene still trains) — just no longer dominant. Result: healthy hand + preserved scene (obj L2 unchanged at 14 mm).

Why the "frozen" shortcut works. Freezing heads caps the effective gradient pathway into backbone features — `∂loss/∂feat = J_head^T · ∂loss/∂output` is bounded by the pretrained J_head's rank. That fixes the same symptom via a different lever (limiting rank of pathway, not magnitude of weight). Frozen is a coarse knob; tuning loss_weight_scene is the precise one.

2-stage training proposal — tested, doesn't help

User question (post-v5): if frozen scene heads + trainable global_blocks + scene losses can adapt pred scene to GT (confirmed: v3 A reaches scene total −0.684, z q95 = 4.55 m, matching all other runs), then do we need to train scene heads at all? Could we do 2-stage training — stage 1 scene-only, stage 2 DCF-only?

Answer to the first half: no, scene heads don't need training.

The scene total reached by v3 A (frozen heads) is the deepest of every config tested, matching or exceeding what unfrozen heads achieve. The backbone has enough capacity (302 M params) to shift features so the fixed pretrained head outputs GT-matching pointmaps. Training scene heads adds parameters but not quality — and for this smoke, actively hurts DCF hand rank (root cause above).

Answer to the second half: 2-stage tested empirically, doesn't pay off.

Why joint training beats scene-only on scene quality

The DCF hand/object losses provide per-point world-space supervision that complements the dense per-pixel scene supervision. The backbone features that encode hand/object position are the same features encoding scene structure — training them together with both signals yields better features than training against scene alone. Concretely:
  • Scene loss supervises dense pointmaps but has weak per-point identity (normalize-first-view=False gives per-view scale invariance).
  • DCF hand/obj loss supervises specific 3D anchor points with known correspondences (MANO vertices, object pose_y transforms), anchoring backbone features at known locations.
  • These signals are complementary: one dense-smooth, one sparse-precise. Joint optimization exploits both.
The 2-stage approach loses this synergy in stage 1, and the scene features are strictly less informative than joint-trained.

Variant worth testing at Phase-3 scale: warmup schedule

Instead of hard stage boundaries, use a continuous loss_weight_scene schedule:

# pseudo-code for proposed scheduler
if step < warmup:      lw = 1.0       # scene-focused warmup
else:                   lw = 0.1       # DCF takes over

total_loss = lw * scene_loss + hand_loss + obj_loss + contact_loss + pointer_loss

This keeps gradient always flowing through both heads (retains the v3 A synergy), but biases early training toward scene convergence. No checkpoint engineering required. A cheap ACP test can compare this against vanilla v5 sw=0.1 to see if warmup buys anything on full 72-frame sequence.

Bottom line for the 2-stage question

2-stage doesn't help on this 5-frame smoke. The joint synergy between DCF and scene losses is real and measurable (~13% scene-loss improvement). The core hand-rank problem was already solved by v5 sw=0.1 (single-stage, unfrozen, scene-weight 0.1) with hand L2 20.6 mm, obj L2 14.1 mm, SVD 0.373.

For Phase 3 full-sequence, the natural extensions worth testing (in order of effort):
  1. Run v5 sw=0.1 on full 72 frames — verify findings hold at 15× scale
  2. A/B sweep loss_weight_scene ∈ {0.1, 0.3, 0.5, 1.0} to find the Phase-3 optimum (the balance may shift with more data)
  3. Warmup-schedule variant (above) — cheap to implement, possibly marginal gain
  4. Only if all above plateau: consider proper 2-stage with stage-2-backbone-trainable-at-low-LR
