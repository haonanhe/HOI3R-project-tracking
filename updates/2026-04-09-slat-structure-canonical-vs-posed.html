2. SLat Encoding Results

Each mesh variant produces a different SLat with different spatial structure:

Mesh variantLatent tokensFeature dim
Canonical T-pose63132
Posed (no orient)40132
Posed full GT64032
Critical finding: token counts differ (631 vs 401 vs 640). The Trellis.2 encoder voxelizes the mesh surface at sparse_resolution=256, producing a variable number of occupied voxels depending on the mesh shape and pose. Different poses occupy different voxel cells, resulting in different spatial grids with different token counts and coordinates.

This means a direct L2 loss between latent features of canonical and posed meshes is undefined — the feature tensors have different lengths and correspond to different spatial locations.

4. Implications for Loss Design

FINDING Latent-space L2 loss cannot drive canonical → posed deformation

Two fundamental reasons:

  1. Token count mismatch: canonical SLat has 631 tokens, posed has 640. No 1-to-1 correspondence for L2.
  2. Spatial grid mismatch: even if token counts matched, the voxel coordinates differ (different mesh surfaces occupy different cells). Comparing features at misaligned spatial locations is meaningless.

FINDING Mesh-level loss is the natural solution

Chamfer distance operates on decoded mesh vertices, which always live in the same 3D space regardless of latent structure. The required gradient path:

canonical_mesh
    ↓
slat_encoder (frozen) → canonical SLat (631 tokens, 32-dim)
    ↓
input_proj (32 → 1024)
    ↓
global_blocks (attend to image visual tokens → learn pose from image)
    ↓
output_proj (1024 → 32)
    ↓
slat_decoder → predicted mesh vertices
    ↓
chamfer_loss(predicted_mesh, posed_GT_mesh)  ← mesh-level supervision

The global_blocks must learn to modify the canonical SLat tokens (based on image context) such that the decoder produces the per-frame posed mesh. This is the core architectural hypothesis of HOI3R.

CHALLENGE Token count changes through the pipeline

The encoder produces 631 canonical tokens. The decoder expects input at the same spatial resolution. If the global_blocks need to produce a posed output with 640 tokens, this requires either:

  • Same-token-count deformation: modify the 631 feature values (not coordinates) to represent posed geometry, relying on the decoder to interpret differently-valued features at the same spatial locations.
  • Token insertion/deletion: the global_blocks would need to change the number of tokens, which dense attention naturally supports (SLat tokens are just extra tokens in the attention sequence).

The current pipeline keeps token count fixed (bridge preserves coordinates). The decoder must interpret the modified features at the canonical spatial grid as representing posed geometry. Whether this is possible depends on how much pose information is encoded in features vs. spatial structure.
