Brainstorm → Design spec

The session started from the user's raw request to "create a worktree for benchmark" with baseline methods (MagicHOI, FP+HaWoR), datasets (HO3D, DexYCB test, HOT3D test), a metric suite, and ideas from four reference papers. A 7-question brainstorming loop locked down:

QuestionDecision
Run FP+HaWoR on HO3D/DexYCB too?Yes — full pipeline on all three datasets.
DCF evaluation source?Inference from checkpoint → same metric code as baselines.
W-MPJPE / camera-space MPJPE coverage?Both, on all datasets (static ⇒ trivially equivalent).
Template-free object handling?SAM 3D Objects reconstructs the mesh; FP+HaWoR and DCF both get template-free variants on all three datasets.
Interaction metrics beyond Contact?Add HOI ADD, HOI ADD-S (WHOLE paper) and CD_h (MagicHOI paper).
Worktree placement?Dedicated .worktrees/feat-benchmark — benchmark code is structurally independent from DCF.
Extra experiments from papers?(1) Occlusion-stratified DexYCB, (3) Zero-shot generalization on HOT3D, (4) Temporal smoothness, (5) Template-based vs template-free.

Architecture (Approach C)

Two-stage pipeline with registry dispatch + unified metric backend:

CLI (benchmark.py)
   ↓ --method --dataset --variant --stage {run,eval,all}
Registry (registry.json) —— Adapters ——→ predictions.npz / gt.npz / metadata.json
                                                      ↓
                                              Unified Evaluator
                                        hand / object / hoi / temporal
                                                      ↓
                                           summary.json + per_sequence.json
                                                      ↓
                                        aggregate.py + latex_export.py
                                                      ↓
                                           NeurIPS-ready tables

Key property: every method produces identical standardized output, so the evaluator is method-agnostic. All three methods (MagicHOI, FP+HaWoR, DCF) are scored with the exact same metric code — essential for reviewer-defensible fair comparison.

Unified metric suite

Hand metrics (benchmark/metrics/hand_metrics.py — 336 LOC)

  • MPJPE — root-relative
  • PA-MPJPE — per-frame Procrustes alignment (SVD)
  • W-MPJPE — world, aligned using first 2 frames
  • WA-MPJPE — world, aligned using all frames
  • AUC — 0–50mm integrated
  • CD — bidirectional Chamfer on hand mesh
  • Accel — mm/frame²

Object metrics (object_metrics.py — 254 LOC)

  • ADD — corresponding points after pose transform
  • ADD-S — symmetric (nearest-neighbor via cKDTree)
  • CD, F5, F10 — on vertex clouds
  • ACC — object centroid acceleration

Interaction metrics (hoi_metrics.py — 323 LOC)

  • Contact F1 / Precision / Recall — per-hand-vertex binary contact
  • CD_h — hand-root-relative Chamfer (MagicHOI)
  • HOI ADD / HOI ADD-S — object ADD after aligning via predicted hand trajectory (WHOLE)

Temporal metrics (temporal_metrics.py — 58 LOC)

  • jitter_ratio — std(accel) / mean(accel)
  • per_frame_acceleration — for smoothness curves
Applicability rules are enforced by the evaluator: ADD / ADD-S / HOI ADD only apply when metadata["object_model_source"] == "gt_cad"; Contact metrics only when contact_prob is present; non-applicable metrics return null in the JSON (never a dummy number).

Bugs found & fixed during implementation

#BugCaught byFix
B1 Wrong coord convention multiplication: plan's opengl_to_opencv used right-multiply (pose @ flip) — does not negate translation Y/Z. Implementer subagent during Task 3 Changed to left-multiply (flip @ pose), the correct convention for camera extrinsics conversion.
B2 Chamfer test assumption wrong: test_offset_verts assumed translating random points by uniform vector yields CD = vector magnitude. Fails because dense random points allow non-corresponding NN matches closer than the offset. Implementer subagent during Task 5 Rebuilt test with well-separated 10×10×10 grid (spacing 111mm >> offset 7mm).
B3 Directory parsing breaks on method names with underscores: split("_", 2) on fp_hawor_ho3d_template_based gave method=fp, losing the rest. Implementer subagent during Task 16 Changed to rsplit("_", 2) → method=fp_hawor, dataset=ho3d, variant=template_based.
F1 manopth not installed in human3r env, but smplx is. Both implement MANO but smplx outputs meters (vs manopth mm). Implementer subagent during Task 8 All loaders use smplx.create and multiply by 1000 to match the mm convention. Dataset-specific use_pca / flat_hand_mean flags match each toolkit's official MANO usage.
F2 MagicHOI is not ready-to-run: user flagged that the previous "MagicHOI HO3D results" in the baseline bundle were never actually produced. MagicHOI's sequence_config.py has hardcoded entries and its pipeline needs COLMAP + HaMeR + inpainting preparation. Workflow change: Codex delegation

Established a new collaboration pattern for this project going forward:

  • Code writing / debugging → delegated to codex:codex-rescue subagent
  • Everything else (planning, research, reads, tests, git, docs, config, coordination) → handled by Claude directly
  • After Codex returns: Claude reads the diff and reviews for spec compliance + quality
  • Codex unreachable: Claude falls back to writing the code itself
  • Follow-up turns: use --resume flag (routes to --resume-last) to continue the same Codex thread cheaply

Rationale: Codex handles code-writing cheaper under the ChatGPT usage budget; Claude's tokens are best spent on review, coordination, and non-coding work.
