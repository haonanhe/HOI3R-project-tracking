Key Discoveries

FINDING Approach-phase hand spike is causal-decoder cold-start

On seq 20200709_141754, both exp_a and exp_b show a hand L2 spike in frames 0–9. exp_a memorized the mid-sequence (8.46 ± 1.39 mm) yet its approach is 24.69 ± 11.63 mm (max 50.04 mm at frame 0). Memorization doesn't fix it → not a data / convergence issue; it's the model's causal-decoder warming up its temporal context. Motion (27.7 → 9.5 mm/frame), hand-object distance (323 → 86 mm), hand bbox area (3885 → 6143 px²), and visibility (60.5% at frame 0 → 100% by frame 3) all co-vary with the spike, but the exp_a evidence rules out data as the driver.

Follow-on experiment (seed for next spec): prepend K=10 duplicated / jittered copies of frame 0 as a warmup, then evaluate the real sequence. If spike disappears → H1 (cold-start) confirmed. If persists → H3 (motion) or H2 (visibility).

FIX 8×8 DDP OOM → expandable_segments env var

Single-GPU memory probe reported 67.65 GiB peak at bs=8 (11.5 GiB headroom on 80 GB H100). First B.00 (pt-6z1krbfg) OOMed at 75.14 GiB — DDP gradient buckets + NCCL buffers + allocator fragmentation added ~7.5 GiB not captured by the single-GPU probe. Fix: PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True in the ACP env — the "reserved but unallocated" 9.37 GiB pool that triggered the OOM was largely fragmentation. B.00-v2 (pt-05jjn033) with same bs=8 config runs cleanly.

FIX memprobe build_args drifted from trainer — object_loss_mode missing

After B.1 added --object-loss-mode to scripts/dcf_dataloader_train.py, scripts/dcf_memory_probe.py crashed with AttributeError: 'SimpleNamespace' object has no attribute 'object_loss_mode' because compute_batch_loss reads it. Fixed at cf53855 by backfilling the default in probe's build_args(). Lesson: any new CLI flag read in compute_batch_loss must also be added to the probe's hand-maintained SimpleNamespace.

FINDING 8×8 pays per-epoch overhead: 1 step/epoch on a 100-seq dataset

At 8 GPU × bs=8 (global batch 64) on a 100-sequence dataset with drop_last=True, there is only 1 optimizer step per epoch. Workers respawn each epoch (persistent_workers=False) adds ~14 sec of overhead per epoch; net step-time is ~24 s vs the probe's 10.83 s. Projected B.00 wall doubles from ~6 h to ~13 h. No fix attempted this session — accept the wall-time cost. Future improvement: set oversample_manifest to multiply epoch length, or enable persistent workers.

FINDING bs=8 at N_o=1024 survives with expandable_segments

Pre-session expectation was that bs=8 / N_o=1024 would OOM even with fragmentation fix. Probe (pt-jll8rmym) measured 72.45 GiB / 6.73 GiB headroom at bs=8, and full run pt-2b5xrw2h passed epoch 1 cleanly at 30 s/step. Keeps B.5 apples-to-apples with B.00 at effective batch 64 rather than de-rating to bs=6 / effective 48.
