2026-04-15 · DCF Scene-Loss Audit vs STream3R Official

Finding: point_loss missing Finding: single-view iteration Finding: 65535mm sentinel not filtered Reference: stream3r/loss/losses.py:1762–1803

User question: are DCF scene losses calculated strictly following STream3R's official code, including how GT point clouds and camera are loaded from DexYCB? This page does three things: (1) enumerates every kwarg of depth_loss / camera_loss / point_loss in the official CausalLoss.compute_loss and diffs them against our training script, (2) traces GT construction through stream3r/dust3r/datasets_cut3r/base/base_multiview_dataset.py and compares with our DexYCB loader, (3) visualises the GT supervision on tracked frame 67 (input, depth heatmap, valid mask, back-projected pointmap) alongside the pretrained STream3R model's prediction on the same frame.

Bottom line — we are mostly following STream3R, but with three non-trivial deviations:
  1. We skip point_loss on the global pointmap. Official totals loss_pts3d_conf + loss_pts3d_grad + loss_depth_conf + loss_depth_grad + loss_camera. We only total loss_depth_conf + loss_depth_grad + loss_camera — missing about 40% of the scene-loss signal. Fix is trivial: the model already emits preds["world_points"] + preds["world_points_conf"], and GT pts3d is a one-line unprojection of GT depth through depthmap_to_absolute_camera_coordinates.
  2. We don't filter the uint16 65535mm sentinel. DexYCB depth PNGs encode "no reading" as 65535, which becomes 65.5 m after /1000. Our valid_mask = depth > 0 accepts it. STream3R's valid_range=0.98 quantile filter partially hides it, but a handful of 65m outliers warp the normalization denominator before filtering.
  3. We iterate one frame per step. STream3R was trained on multi-view batches where camera_loss has relative-pose targets. Our S=1 invocations give every view an identity relative-pose target, so camera_loss contributes essentially zero signal — even when loss-weight-scene=1.0. (Depth loss is still valid per-view; only the camera-loss term is degenerate.)

Canonical STream3R call site

From CausalLoss.compute_loss at stream3r/loss/losses.py:1762. GT comes from get_pts3d_from_views (line 369), get_depth_from_views (line 407), get_camera_from_views (line 429). Each gt_view must carry pts3d (world-space pointmap, derived automatically from depth+intrinsics+camera_pose by base_multiview_dataset.py), depthmap, valid_mask, camera_pose (cam2world, 4×4), camera_intrinsics (3×3).

gt_pts3d_global, valid_mask_global = self.get_pts3d_from_views(gts, key_word="pts3d", **kw)
gt_depth, valid_mask_depth = self.get_depth_from_views(gts, **kw)
gt_extrinsics, gt_intrinsics, image_size_hw = self.get_camera_from_views(gts)

pred_pts3d_global, pred_conf_global = preds["world_points"], preds["world_points_conf"]
pred_depth, pred_depth_conf = preds["depth"], preds["depth_conf"]

loss_pts3d_global = point_loss(
    pred_pts3d_global, pred_conf_global, gt_pts3d_global, valid_mask_global,
    gradient_loss=self.gradient_loss, temporal_matching_loss=False,
    all_mean=True, valid_range=0.98,
    normalize_pred=True, normalize_gt=True, normalize_using_first_view=False)

loss_depth = depth_loss(
    pred_depth, pred_depth_conf, gt_depth, valid_mask_depth,
    gradient_loss=self.gradient_loss, temporal_matching_loss=False,
    all_mean=True, valid_range=0.98,
    normalize_pred=True, normalize_gt=True, normalize_using_first_view=False)
gt_pts3d_scale  = loss_depth["gt_pts3d_scale"]
pred_pts3d_scale = loss_depth["pred_pts3d_scale"]

loss_camera = camera_loss(
    pred_pose_enc_list, gt_extrinsics, gt_intrinsics, image_size_hw,
    loss_type="l1",
    gt_pts3d_scale=gt_pts3d_scale, pred_pts3d_scale=pred_pts3d_scale,
    pose_encoding_type="relT_quaR_FoV")

pts3d_loss = (loss_pts3d_global["loss_conf"] + loss_pts3d_global["loss_grad"]
            + loss_depth["loss_conf"]       + loss_depth["loss_grad"])
total_loss = pts3d_loss + loss_camera["loss_camera"]

The GT pts3d tensor is built automatically by base_multiview_dataset.py:570–574 via depthmap_to_absolute_camera_coordinates(depthmap, camera_intrinsics, camera_pose), which unprojects every pixel and then transforms to world (first-view) space. valid_mask = (depth > 0) & finite(pts3d).

GT loader structure — ours vs base_multiview_dataset.py

STream3R's base_multiview_dataset.py:533–574 explicitly asserts that pts3d is NOT present on the view dict and builds it in-place from depth+intrinsics+camera_pose. Every loader in stream3r/dust3r/datasets_cut3r/ follows that contract.

# official — base_multiview_dataset.py:570
assert "pts3d" not in view
assert "valid_mask" not in view
pts3d, valid_mask = depthmap_to_absolute_camera_coordinates(**view)
view["pts3d"] = pts3d
view["valid_mask"] = valid_mask & np.isfinite(pts3d).all(axis=-1) & (not blk_img_flag)

Our loader in dcf_full_sequence_train.py:load_full_sequence_dexycb emits depth_per_frame, depth_valid_per_frame, intrinsics_518, extrinsics — but never materialises pts3d. That's the data-side manifestation of the same 40%-missing-signal bug: because the loader doesn't carry pts3d, the training step can't call point_loss.

Proposed fixes (ordered by signal gain)

  1. Add point_loss to the scene-loss call (~40% extra signal). Build gt_pts3d once in load_full_sequence_dexycb via depthmap_to_absolute_camera_coordinates using depth_per_frame, intrinsics_518, extrinsics. At loss time pass preds["world_points"] + preds["world_points_conf"] and add loss_pts3d["loss_conf"] + loss_pts3d["loss_grad"] to the scalar.
  2. Filter the 65535 sentinel at depth ingest. In load_full_sequence_dexycb, depth_m[depth_mm == 65535] = 0.0 before the nearest-interpolate. The median-based normalization is sensitive to outliers because valid_range=0.98 kicks in after median scaling.
  3. Tighten valid mask. After building pts3d, set valid = (depth > 0) & isfinite(pts3d).all(axis=-1). Mirrors base_multiview_dataset.py:573.
  4. Multi-view batching. Sample S=4 frames per training step and stack into [1, S, 3, H, W] so camera_loss has non-identity relative-pose targets. This requires nothing from DexYCB (the GT extrinsics are still identity per frame for a static camera, but STream3R's pose encoding operates on predicted relative poses anchored by first-view, so the supervision signal comes from driving pred relative poses to identity — which is a non-trivial target when the feature backbone sees 4 different RGB frames).
  5. Switch image resize to Lanczos + cap-crop to square. Match _crop_resize_if_necessary. Not expected to change metrics on this smoke, but closes the last known divergence.

Fixes #1–#3 are small changes to scripts/dcf_full_sequence_train.py and stream3r/data/dexycb_toolkit_bridge.py. Fix #4 is a batching restructure worth a separate diff. Fix #5 is cosmetic.
