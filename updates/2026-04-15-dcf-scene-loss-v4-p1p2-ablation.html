v4 ablation Branch: feat/dcf Commit: 44446b7

Targeted fixes for the two follow-up bugs flagged on the v3 page:

  • P1: loss_pts3d_grad / loss_depth_grad were pinned at constants (0.0130 / 0.0389) in v1/v2/v3 — the multi-scale gradient loss was insensitive to pred changes. Fix: --scene-grad-mode none passes gradient_loss=None to depth_loss / point_loss.
  • P2: Unfrozen runs collapsed pred_hand to a line (SVD s3/s1 ≤ 0.07). Fix: --hand-loss-type chamfer uses bidirectional Chamfer (forces coverage of GT cloud).

Three unfrozen runs, same 5 tracked frames × 200 steps × num-views=4 × lr-scene-heads=1e-5. Comparison against v3 A (frozen control) and v3 B (unfrozen baseline).

TL;DR — each fix works for its own target but they don't compose cleanly:
  • P1 alone (grad=none): hand L2 18.2 mm, obj L2 15.8 mm (both best). But hand still a line (SVD 0.071).
  • P2 alone (Chamfer): hand SVD s3/s1 = 0.323 (near GT 0.40). But hand L2 jumps to 68.1 mm, obj to 89.4 mm.
  • P12 combined: SVD 0.216 (healthy), hand L2 39.6, obj L2 117.1 mm (worse on obj).
Recommendation: land P1 unconditionally (free L2 win). P2 needs refinement — Chamfer shouldn't replace L1 for objects (rigid, per-point correspondence via pose_y is meaningful). Test l1+chamfer on hand only; keep L1 for obj.

Why P2 hand L2 is worse despite higher rank: Chamfer finds closest-point matches without enforcing correspondence. Pred can spread across the GT cloud (high rank) while losing the identity mapping pred_i ↔ gt_i that MANO preserves. L2 penalises that identity loss; Chamfer doesn't. So Chamfer recovers shape but sacrifices correspondence.

Interpretation & follow-ups

What P1 taught us

Disabling the static _grad components of scene loss improves hand L2 by 45% and obj L2 by 25% (32.8 → 18.2 mm hand, 12.6 → 15.8 mm obj vs v3 B — wait, obj slightly worse on v3 B because v3 B was already near the optimum on obj). The "dead weight" hypothesis is confirmed: the static 0.0130 / 0.0389 phantom was adding gradient noise that impaired optimisation. Removing it is a pure win on L2.
Recommendation: change --scene-grad-mode default from grad to none.

What P2 taught us

Bidirectional Chamfer recovers hand rank (SVD 0.03 → 0.32) but it's NOT a drop-in replacement for L1 because:
  • MANO hand vertices have ordered correspondence — vertex i of the pred hand should match vertex i of the GT hand (the DCF head consumes per-vertex canonical input). L1 enforces this; Chamfer does not.
  • Rigid objects (via pose_y) also have per-point correspondence. Applying Chamfer to objects (P12) gave obj L2 117 mm, the worst result of any run.
  • Pred cloud can "well-cover" the GT cloud geometrically while being scrambled per-vertex. L2 penalises that; Chamfer doesn't.
Next experiment: try --hand-loss-type l1+chamfer (already implemented, not yet run) on hand only, keeping L1 for object. Or add a cheap rank regulariser (`−det(Σ_pred + ε·I)` or `Frobenius(pred - pred_mean).det()^(1/3)`) on top of L1.

What still isn't understood

  • Why P12 obj L2 = 117 mm. Disabling grad AND using Chamfer simultaneously produced obj prediction that's substantially worse than either individually. Likely: Chamfer on object loses the rigid-body correspondence, and without the grad component pinning things, the optimiser wanders further. But we didn't fully diagnose.
  • Why v4 P1 obj L2 (15.8) is worse than v3 B (12.6). v3 B had the "dead weight" grad — somehow that was helping obj in v3 B. Possibly the grad component acts as a weak regulariser for obj rigid-body fit. Worth understanding before landing P1 as default.

Concrete next steps

  1. Land P1 as default after one more confirmation run: v4 P1 on a second sequence, verify the L2 improvement replicates.
  2. Test l1+chamfer hand-only: split the --hand-loss-type into --hand-loss-type + --obj-loss-type so Chamfer can target hand while obj stays L1.
  3. Investigate static-grad cause: trace why gradient_loss_multi_scale returns the same value for very different predictions. Either fix gradient_loss_multi_scale or remove the multi-scale loop.
  4. Phase-3 A/B on full 72-frame sequence with the settled config.
