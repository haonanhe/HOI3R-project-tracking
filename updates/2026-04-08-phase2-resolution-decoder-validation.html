Changes Validated

1. Sparse resolution 8 → 256

Updated three config files to match Trellis.2 pretrained decoder's native resolution (256). The encoder downsamples 4x to 16³ bottleneck; the decoder upsamples back to 256. Resolution 8 collapsed hand/object to single-token representations.

2. Global blocks (already unfrozen)

Verified that global_blocks already have requires_grad=True when freeze_stream3r_encoder=true. Only frame_blocks and patch_embed are frozen. No code change was needed.

3. Training-mode decoder path

Modified _decode_slot to call the base SparseUnetVaeDecoder.forward() directly in training mode, bypassing FlexiDualGridVaeDecoder (which requires gt_intersected with matching spatial coordinates). The base decoder returns (h, subs_gt, subs) for supervision. Mesh extraction uses predicted intersected flags.

4. Direct mesh supervision losses

Enhanced _compute_trellis_mesh_loss with three new terms: intersected logits BCE, subdivision prediction BCE, and latent reconstruction MSE. Config weights added: intersected_weight=1.0, subdiv_weight=0.5, latent_recon_weight=1.0.

5. Triton flex_gemm backward pass fix

Three-part fix: (a) convert decoder to fp32 via convert_to_fp32() to resolve fp16/fp32 dtype mismatch in tl.dot; (b) switch to explicit_gemm algorithm to avoid Triton 3.x tl.cdiv signedness bugs; (c) patch torch.flip for uint32 tensors in flex_gemm.

Verdict

Resolution 256 produces geometrically meaningful meshes with vertex counts matching the Trellis.2 reference (170,450 vs 170,401 for hand). The trainable decoder path works end-to-end with stable gradient flow through 200 optimization steps. Resolution 8 collapsed all geometry to ~100-200 vertices.

Next steps: (1) Run full closed-loop training with STream3R aggregator, (2) add chamfer distance evaluation against GT mesh, (3) profile explicit_gemm vs potential Triton kernel fixes for speed.
