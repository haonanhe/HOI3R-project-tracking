Key discoveries

FINDING Discussion notes reframed v1 as a video-conditioned canonical 4D correspondence field

User's hoi3r_4rc_dcf_discussion_notes.md argues the 4-head HandGeom/ObjGeom/HandMotion/ObjMotion topology is the wrong mathematical object — it can learn mutually inconsistent {Φ(q,t), Φ(q,τ), Δ(q,t→τ)}. Single-Φ Φ(q,t) makes Δ derived: Δ(q,t→τ) := Φ(q,τ) − Φ(q,t), so cycle/transport composition holds by construction. This explains the cl=4-train→cl=16-eval 10× regression and the motion-grounding null result. v1 commits to the doc's §22 minimal scope (single-Φ + cycle + object Procrustes) plus encoder LoRA + scene loss. Hand LBS+residual deferred to v2.

FINDING Pose-query mechanism = repurposed motion head

Discovered that the legacy geom heads (HandGeomHead, ObjGeomHead) are tiny per-token MLPs with NO cross-attention to image features and NO time conditioning. The only mechanism for a query to look at frame-t image features is the motion head's CrossBlock + AdaLNBlock. So 'single-Φ pose-query' isn't a new architecture — it's a renamed motion head reinterpreted as per-frame Φ(q,t) decoder, with context = frame-t patches and time-cond = frame-t time token (no source/target asymmetry). AdaLN time-conditioning is inherited; nothing new to design.

FINDING Hand 21 joints, not vertex samples

Predicting 128 free xyz canonical samples per hand slot is intractable without articulated structure. v1 reduces hand identity space to 21 MANO joints — 6× smaller, well-supervised by GT, eligible for 2D-projection auxiliary loss. Vertex recovery is post-hoc via MANO joint-fit at eval (LBFGS over θ with β frozen). Lever-d (commit 269531b) already plumbed 21 joint GPS tokens; v1 elevates them from auxiliary to primary.

FIX PyTorch3D unavailable in human3r env

Spec recommended pytorch3d.ops.corresponding_points_alignment for weighted Kabsch; verified it's not installed. Plan rolls our own via torch.linalg.svd (recipe in Task 4 of plan). Reference impl at stream3r/loss/dcf_losses.py:199- (unweighted).

FINDING Object branch losses lock to exactly 2 terms

L_final (Procrustes-projected output vs GT, weight 1.0) + L_raw_nll (per-point heteroscedastic NLL on noisy correspondences vs GT, weight 0.1). σ does double duty: NLL-supervised AND used as Procrustes weight (1/σ²). Symmetric YCB objects filtered (quotient loss deferred to v1.1). No explicit rotation or translation loss — point-on-surface loss already pins them.
