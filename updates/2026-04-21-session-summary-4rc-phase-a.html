Key discoveries

FINDING

Arc forward API: list[dict] with img key, not a raw tensor

The plan's A5 assumed model(frames) with a (B,T,C,H,W) tensor. Actual API: model([{"img": tensor_BCHW}, ...]) where T is the list length. Per-frame output dict has 9 keys (pts, conf, track, conf_track, track_multi, conf_track_multi, track_query_idx, extrinsic, intrinsic). Recorded in docs/third_party_4rc_note.md; carried into Phase B.

FINDING

HF repo Luo-Yihang/4RC has NO config.json

Hub siblings are .gitattributes / LICENSE / README.md / model.safetensors only. Arc.from_pretrained("Luo-Yihang/4RC") fails with the HF .no_exist sentinel unless a fabricated minimal config.json sits in the snapshot dir. All six Arc.__init__ kwargs have defaults, so a default config matches the checkpoint arch (verified against 1.52 B param count).

FINDING

Upstream batch-dim bug in _postprocess_output

extrinsic[4,4] and intrinsic[3,3] return without a B dim — only the last batch element's camera is preserved. Harmless at B=1 (our usage). Phase B's encoder wrapper either forces B=1 or bypasses _postprocess_output.

FIX

Depth distortion from aspect-squishing (v1 → v2)

First-pass smoke center-cropped 480×640 → 480×480 then resized to 504×504 — a ~25% horizontal content loss + stretch. 4RC was trained on aspect-preserving load_images(size=512, patch_size=14) input, which yields 504×378 for 480×640 DexYCB frames. Raw model(views) in fp32 also skipped the bf16 autocast that inference() applies. v2 smoke (f8ad2c8) uses the README's load_images + inference(dtype="bf16-mixed") path verbatim. Depth range tightened from 0.37–2.86 m (broken distribution) to 0.47–2.80 m (p5/p95 around median 2.03 m — geometrically coherent).

FINDING

huggingface.co unreachable from debug container

Returns [Errno 101] Network is unreachable. Workaround: HF_ENDPOINT=https://hf-mirror.com — Tsinghua-hosted read-only mirror of the hub. All future HF calls from the debug container need this. Memory reference_hf_mirror_debug_container.md captures the pattern + known-good cache paths.

FIX

Publish via Git Data API single commit, not 16 Contents API commits

User directive: when uploading multiple files to the tracking repo, use the Git Data API (blobs → tree → commit → ref) so N files land in ONE commit. The Contents API PUT-per-file approach creates 1 commit per file (16 commits for a typical page+assets set). The robot_arm publish used the Data API pattern end-to-end (7 API calls: 4 blobs + 1 tree + 1 commit + 1 ref). Captured in updated reference_publish_html.md.

FINDING

Viser visualizer: flatten predictions.pt → result.npz

The viser CLI expects the flat pred_<i>_<k> / view_<i>_<k> schema produced by inference.py's save_npz. Our v2 smoke saves a predictions.pt in PyTorch dict format — a 10-line flattener converts it. Also: viser silently auto-increments port (8020 → 8021) if the requested port is taken, so ssh -L 8020:8020 ends up pointing at nothing. Always pkill -f viser_visualizer_track on remote before starting.
