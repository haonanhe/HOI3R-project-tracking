Key findings

WIN Closing the train/eval H/W mismatch shrinks the gap ~6×

Previous eval at dataloader-default 518×518 on 16 frames produced hand/obj L2 of 88.6/50.7 mm (E1). Re-running with --full-sequence --fixed-h 378 --fixed-w 504 — matching 4RC's inference recipe and the training H/W — the numbers collapse to 14.7 / 5.6 mm. Every cell improved by 5–10× on both metrics. The old "large train→eval gap" was almost entirely domain shift from H/W, not actual generalization failure. This also explains why the ablation looked so uniformly bleak at 518 square — the rectangular 378×504 inference the 4RC backbone was trained on doesn't survive center-crop to 518².

WIN Scene-loss is still the strongest single intervention

pt-cwhz5bvz (E1 + --scene-loss-weight 0.1) wins every metric: train hand 8.31 / obj 3.93, eval hand 12.39 / obj 4.79 mm. Beats the vanilla E1 LoRA baseline by 16% on hand and 14% on obj at eval, and is the only cell that crosses below obj 5 mm. The aux depth+camera loss is cheap (single-forward shared feats, no extra encoder pass) and consistent with the hypothesis that it prevents the LoRA from collapsing into a motion-only shortcut.

REGRESS All 4 LoRA cells train WORSE than the frozen baseline

Frozen-encoder baseline pt-9fa652zu (logs/hoi3r_stage1_overfit/train.log, parsed today after a directory-mapping correction) reaches best-of-last-20 hand 5.08 / obj 3.57 mm, clearing the exit bar (hand < 5 is marginal; obj < 3 is the only miss). Every LoRA cell regresses on both metrics: E1 8.72/5.00 (+72%/+40%), E4 9.66/5.57 (+90%/+56%), E1+scene 8.31/3.93 (+64%/+10%), E1+patchctx 9.17/4.62 (+80%/+29%). Scene-loss has the smallest obj regression but still misses. LoRA r=16 α=32 doesn't help at this regime — it adds capacity that memorizes training noise instead of improving the geometry/motion reconstruction the baseline already does well.

REGRESS Combined (shared-weight) motion head is the weakest cell

pt-beaua22b eval obj 8.59 mm vs E1's 5.60 — 1.5× worse, and eval hand 18.27 vs 14.72 — 1.2× worse. The gap is smaller than it looked at 518² (2× obj) but CombinedMotionHead still underperforms the separate-head baseline. Weight-sharing lets hand gradients dominate the object branch at training time; no evidence it helps and clear evidence it hurts on this regime.

REGRESS Patches-only cross-attn is a wash

pt-e9dqtgv7 (scene-loss + patches-only) sits at eval 15.22/5.19 — worse than scene-loss alone (12.39/4.79) by small margins. Restricting the motion-decoder cross-attn context to image patches (dropping other-frame hand/obj query tokens) doesn't help. At 518² the regression looked catastrophic (79/90 vs 58/32); at 378×504 it's marginal. Other-frame query tokens don't pollute, but they don't clearly help either at this scale.

GATE Plan phase-1 decision: NO-IMPROVEMENT

Per plan 2026-04-23-motion-grounding-ablation-plan.md §Task 8: every LoRA cell trains worse than the frozen baseline (5.08 / 3.57). Decision: NO-IMPROVEMENT → STOP. The full-sequence eval at training-matched H/W does show scene-loss helps within the LoRA family (E1+scene beats E1 by ~16%/14% on full-seq eval), but the LoRA family itself is a regression vs frozen. Re-diagnosis required before any more ablation cells — e.g., revisit whether the scene-loss anchoring alone (without LoRA) would help the frozen baseline, or move to multi-sequence training where LoRA's generalization argument actually has teeth.

FINDING Baseline eval still blocked by architecture drift

pt-9fa652zu's checkpoint predates the 2026-04-23 motion-head rewrite (commit 58532a2: HandMotion/ObjMotion now mirror 4RC with AdaLNBlock+CrossBlock and LayerScale) and the 2026-04-22 handedness two-slot fix (commit 7cdd0f5a: e_hand is now [2, 1536]). The state-dict keys don't match current HEAD, so running hoi3r_eval.py on this checkpoint requires a historical code checkout — blocked per feedback_single_worktree.md. Fix: submit a fresh frozen-encoder run at current HEAD so the baseline has a directly-comparable full-sequence eval + v3 render.

Caveats

  • Train/eval H/W mismatch — now closed: re-eval uses --full-sequence --fixed-h 378 --fixed-w 504 to match training and 4RC's inference recipe. The prior round's 518×518 / 16-frame numbers are kept in the table below for reference but are no longer the headline number.
  • Single-sequence overfit: all 4 cells train on one clip (subject-01 seq-01). Evaluating on the same clip at the full 63 tracked frames measures sequence-length generalization and trajectory coherence, not test generalization. Multi-seq s1_test numbers require a separate plan (out of scope).
  • No rotating MP4: v3 render emits 3 camera-view MP4s + interactive Plotly; no world_rotating_slow.mp4 under the default flags. Plotly scatter covers rotation via user drag.
  • Debug-container MIG quirks: subagents hit the NVML allocator assert on torch.load(map_location=cuda) twice. Working fix per agent report: PYTORCH_NVML_BASED_CUDA_CHECK=0 + PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True. One agent additionally autocast the forward to bf16 to fit a 40 GB MIG slice — then cast predictions back to fp32 for the renderer.
  • Training log resolution: 150 entries over 3000 steps = log every 20 steps. Best-of-last-20 is actually best-of-last-400-steps; treat as a smoothed estimate, not a per-step minimum.
  • Baseline eval blocked: pt-9fa652zu's model_final.pt was produced before the 2026-04-23 motion-head rewrite (58532a2) and 2026-04-22 handedness fix (7cdd0f5a), so its state-dict keys don't match current HEAD. Eval on the baseline requires either a historical checkout (unsafe on the single feat/dcf worktree) or a fresh retrain at current HEAD. Baseline training metrics remain valid because history.json is just numbers, not tied to model layout.
  • Directory-mapping correction: pt-9fa652zu's training data actually lives at logs/hoi3r_stage1_overfit/train.log (DDP 8-GPU, 2880 steps). An earlier probe hit logs/stage1_r3_cached_3k/ — a different 1-GPU cached-encoder run — and briefly mis-reported pt-9fa652zu as 10.13/9.90. Caveats and next steps

  • Both jobs are single-sequence overfits (subject-01/20200709_141754); generalization is not measured.
  • 4 levers bundled together — can't attribute the win to a specific lever without further ablation. The CONTROL pins down "LoRA + scene-loss + scene-geom-unfreeze" as a group, but doesn't disentangle within that group.
  • Approach-spike still dominates per-frame error in both runs. Possible follow-ups: higher approach-oversample factor, motion-magnitude-weighted loss, or a temporal curriculum that upweights early frames.
  • Joint anchor (Lever D) shows clean signal: BUNDLED 5.34 mm mean vs CONTROL 16.12 mm. The 21 MANO joint tokens are productively trained on top of the 256 vertex queries.
