Key discoveries

FIX Handedness latent bug fix: 50% of DexYCB training was using wrong-sided GPS

Pre-7cdd0f5 DexYCB16FrameClip silently dropped slot_index, so left-hand sequences (499/1000 seqs) trained on right-hand MANO eigvecs with the shared e_hand embedding. Fix: 6-file surgery (loader → encoder e_hand (2,1536) → query MLP → loss mask). Numerically proved left-hand MANO is exactly flip_x of right-hand (max diff 0.0, Laplacian identical) — reuses the right-hand GPS after x-flip, no need to recompute. All Stage-1 H1–H4 and earlier Stage-2 smokes predate the fix and are tainted.

FINDING Forward pass is not the bottleneck — DataLoader is 62% of step wall-clock

On 4×H100 DDP, cl=16 bs=1: dataloader 3.36 s / fwd 0.57 s / bwd 1.45 s / opt 10 ms / log 129 ms. Total wall 5.4 s but GPU busy only 2 s. Cause: num_workers hardcoded to 2, each worker doing 16× (JPEG decode + LANCZOS 518² + depth PNG + depthmap_to_absolute_camera_coords) per clip. Fix: --num-workers 8 --prefetch-factor 4 + persistent_workers (shipped in 5cc5faf).

FIX Pose rot_err 156° was a reference-frame bug, not a model error

4RC Arc.pose_enc returns cam-to-world RELATIVE TO FRAME 0. DexYCB GT extrinsic is absolute world-to-camera. Comparing pred vs GT without a frame-0 anchor leaked the world offset into the geodesic angle. Frame-0-relative RRA/RTA (pred·pred[0]⁻¹ vs GT·GT[0]⁻¹) drops rot_err from 156° to 0.002° on frame 0 and ≤0.07° on all 8 frames. The residual 0.06° drift is the pristine 4RC per-frame noise floor on a static camera. Prior publish pages that cited 156°/165° as a model error need retraction.

FINDING Encoder gradient checkpointing cuts 48% of per-rank GPU memory

Wrapping vit.process_attention in torch.utils.checkpoint for the 27 post-alt_start blocks (those carrying a trainable grad path) drops per-rank peak from 28 → 14.7 GiB at bs=1 cl=4. Blocks 0-12 skipped: requires_grad=False there, autograd already discards. Plus 131 MiB/rank saved by CPU-resident scene GT (never read in Stage 1 loss). Net: bs=8 cl=4 projected ~44 GiB (was OOM at 75.3). bs=16 cl=4 also plausible at ~65 GiB but unverified.

FINDING 4RC → HOI3R motion-head transfer was 56/166 due to structural mismatch — now 168/168

Pre-58532a2 HOI3R motion heads transferred only Q/K/V + MLP weights (56 of 166 params). AdaLN style, pre-norms, LayerScale, cross-attn normy were ALL different from 4RC MotionDecoder. 58532a2 rewrites HandMotion/ObjMotion to mirror 4RC exactly: CrossBlock + AdaLNBlock, pre-norms, LayerScale γ=0.01, DiT-style AdaLN (6×D: shift/scale/gate × {MSA, MLP}), normy(context) pre-norm. Rename fn collapsed from ~120 lines to 2 regex subs. 168/168 transferable (4 left random = delta_proj + uncert.proj, semantically head-specific).

FAIL Ablation: 4RC-init beats random (as expected), but NEW arch did NOT beat OLD arch at 3000 steps

Three-arm overfit on subject-01/20200709_141754. Arm A (random, old arch, bs=4): final hand 12.6 / obj 13.8, median-20 10.1 / 7.1. Arm B (4RC-init, old arch, bs=4): 10.7 / 8.9, median-20 9.2 / 6.0. Arm B′ (4RC-init, NEW arch, bs=8): 10.7 / 9.9, median-20 10.8 / 10.0. B > A holds (4RC-init wins). But B ≥ B′ on median-20 — arch-match did NOT cleanly outperform. Confounds: bs=8 vs bs=4, LayerScale γ=0.01 slow-start, or 3000 steps is just too short to see the asymptotic payoff. Verdict: use new arch for Stage 2 (paper-faithful + 168/168 transfer), but don't assume single-seq arch-match win; Stage 2 multi-seq is the real signal.

FINDING Renderer: interactive Plotly + RGB-sampled scene + sparse FPS tracks is the best viz we've shipped

v3 page at /updates/2026-04-23-renderer-v3.html: 4 views — scene-pred-only (no RGB bg), scene-bg hand+obj (no tracks), scene-bg hand+obj with tracks, Plotly interactive 3D with 16-frame slider. Scene uses per-point RGB sampled bilinearly from the input frame (via projected pixel coords). Hand/obj tracks are FPS-subsampled to 12 + 20 (from 128 + 256) for legibility. Plotly markers 1.5 for hand/obj, 1.8 for scene; scene alpha 0.85 so RGB shows. No viser, no pyrender — pure matplotlib + Plotly CDN.
