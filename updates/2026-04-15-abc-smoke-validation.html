This page is a corrected republication. The earlier version (taken down) had two errors: (1) Task A used a pseudo-GT distillation approach for scene loss instead of real DexYCB depth/camera supervision, and (2) the 3D scatter plots mis-labeled predicted vs ground-truth traces. Both issues are fixed in this version. All data below comes from fresh ACP runs with corrected code.

Scope caveat: These are 5-frame / 3-to-5-step correctness smokes only. Training did not converge. Metrics are noisy and should not be interpreted as reconstruction quality. The goal is to verify that each code path runs without errors and produces finite, non-zero losses.

Task A — Real Scene Loss

What changed

The previous (incorrect) version used a pseudo-GT distillation module that computed scene loss from teacher-model outputs. This has been replaced with real DexYCB supervision: depth_loss + camera_loss from stream3r/loss/utils.py, using ground-truth depth maps (aligned_depth_to_color_*.png) and real camera intrinsics/extrinsics from the DexYCB dataset. The pattern follows losses.py:1777 from STream3R.

Code changes

scripts/dcf_full_sequence_train.py: +102/−2 lines — depth loader, loss wiring, CLI flag --scene_loss_weight. The distillation module was deleted.

Per-step scene loss values

step 1: 0.243  |  step 2: 0.475  |  step 3: 0.232  |  step 4: 0.150  |  step 5: 0.257
All values finite and non-zero — real depth supervision is active.

Task B — Camera-View Video

What changed

Added automatic camera-view rendering: after training, the script projects predicted hand/object meshes onto original image frames using the camera intrinsics (scaled to 518×518: fx=499.09, fy=665.05, cx=249.73, cy=268.20) and composites them as an overlay. Outputs individual PNGs per frame and an assembled MP4 video.
