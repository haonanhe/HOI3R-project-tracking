1. Context: The Hand "Curve" Bug

Coming into this session, DCF hand predictions were collapsing to a 1D curve instead of a hand shape. The prior session summary diagnosed hand memorization (20 mm train / 185 mm test) but not the per-point collapse. A fresh investigation traced the root cause:

Diagnosis: Two compounding causes make hand tokens lose their per-point identity after 24 layers of global attention:
  1. Point identity is lost after decoder processing. When canonical_dim=0 (the current state after the previous shortcut-revert), the trajectory MLP receives only the 1024D decoded token. After 24 layers of global attention, initial per-point distinctiveness is diluted. Evidence: canonical_dim=3 gave 11.8 mm hand L2; removing it gave 31–37 mm.
  2. All DCF tokens share RoPE position (0,0). In _process_global_attention, extra_pos = torch.zeros((B, extra_token_count, 2)). Attention cannot distinguish DCF tokens by position.
The MLP predicts a per-frame mean position → across frames this traces a 1D curve (the wrist path).

2. Design: GPS-Based Encoding + Skip Connection

The design choice hinged on a single question: where should geometric identity be injected?

RoPE rejected. RoPE creates relative positional bias between Q/K pairs. If DCF tokens get GPS positions and image tokens get pixel-grid positions, the bias is meaningless (the two coordinate spaces are unrelated). RoPE is only useful when tokens share a coordinate system.

Chosen: GPS as token content + raw GPS skip connection.

LevelMechanismDecision
Encoder inputReplace xyz with 16-dim Laplacian eigenvectors (GPS)Chosen
Trajectory head skipConcatenate raw 16-dim GPS (detached) to decoder outputChosen
RoPE positionReplace (0,0) with GPS-based 2D positionRejected
Fourier-encoded skip128-dim rich features as skipRejected (shortcut risk)

Why raw GPS, not Fourier-encoded GPS, as skip: 16 raw eigenvectors give point identity but not enough information for the MLP to predict a 3D posed position by shortcut — the head is forced to use the decoder's image-integrated features.

Why GPS collapsed in prior attempts: GPS was tried as a learnable encoding. Gradients pushed all embeddings to the same value. Fix: precompute eigenvectors and store as a frozen register_buffer.

5. Key Discoveries

5.1 RoPE and content serve different purposes FINDING

The choice of where to inject geometric identity matters. RoPE biases attention based on relative positions in a shared coordinate space; giving DCF tokens positions from a disjoint space (mesh topology) produces meaningless bias against image pixels. Content-level injection (through the Q/K projection) preserves the model's ability to learn appropriate routing rather than hard-coding spatial relationships that don't exist.

5.2 Raw vs Fourier-encoded skip FINDING

A too-rich skip enables the MLP to memorize canonical→posed mappings during overfit, bypassing the decoder. A too-poor skip (e.g. none) loses point identity. Raw 16-dim GPS is a Goldilocks: enough to disambiguate 778 vertices, not enough to predict posed 3D positions alone.

5.3 Why prior GPS attempts collapsed FIX

Learnable GPS encodings are vulnerable to gradient collapse (all points → zero embedding). Freezing the GPS buffer eliminates this failure mode entirely. Additionally: fix sign convention per eigenvector column once at precompute time to avoid sign flips across runs/batches.

5.4 Object encoding not yet upgraded FINDING

Objects still use raw xyz Fourier encoding with no skip to the trajectory head. Object L2 is already excellent (8.1 mm on fullseq) partly because objects are rigid (single SE(3) transform for all points). Per-object GPS could be an ablation, though the payoff is lower than on the articulated hand.

5.5 HTML generation gotcha FIX

When generating HTML programmatically, json.dumps escapes / to \/. If that string is written as raw HTML inside a <script> closing tag, browsers don't recognize <\/script> as a terminator. The script block is never closed, the generator JS never runs, and all data-driven sections render blank. Always write literal </script> in HTML bodies.

5.6 GitHub Pages publishing via Contents API FINDING

The project-tracking repo's .git history is ~250 MB — cloning takes 5+ minutes. Using GitHub's Contents API (PUT /repos/:owner/:repo/contents/:path) with the GitHub token from macOS Keychain bypasses the clone entirely and publishes in seconds. See reference_publish_html.md memory.
