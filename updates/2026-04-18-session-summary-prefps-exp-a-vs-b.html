TL;DR

The object-grouped-sampler constraint was an implementation artifact. Commit 8700315 moves object FPS into the dataset so mixed-object batches stack into [B, 256, 3] naturally — sampler, loss, batch size, and GPU count can all be held constant. User-proposed follow-up: SAM-3D-Objects as an object shape prior at inference, turning the DCF object head into a pose-only estimator.

Key Discoveries

Multi-seq training itself is the regression FINDING

Identical architecture, bs, L1 loss, scene weights, LR, and step budget. Expanding from 1 sequence to 100 sequences on the same subject / camera / hand shape: hand L2 11.8 → 21.5 mm, obj L2 9.2 → 50.1 mm. The gap cannot be attributed to the sampler, loss, or bs any more — those were all equalized by 8700315. Evidence and plots on the experiment page.

Object failure mode = rigid displacement, not misarticulation FINDING

exp_a chamfer/L2 ≈ 1.56 (hand) and 1.56 (obj) — near-1 means predictions are on the right cloud, mildly displaced. exp_b object chamfer/L2 = 1.22 — even lower ratio despite far higher absolute error — means the whole predicted object cluster is shifted by ~40 mm as a rigid group. Signature of an under-specified object-pose prior averaged across 20 YCB objects.

Pre-FPS dataset refactor lifts the object-homogeneous batch constraint FIX

DexYCBDCFDataset.__getitem__ now emits object_query_points: [N_o=256, 3] per sample alongside the full canonical mesh. Because N_o is constant, dcf_collate_fn stacks mixed-object samples into [B, 256, 3] without needing ObjectGroupedBatchSampler. DCFBranch.prepare_extra_global_tokens and STream3R.forward prefer object_query_points when supplied; the old FPS-on-batch[0] path remains as fallback. Added --oversample-manifest N so single-sequence × bs>1 is feasible via per-call random frame resampling.

Memory probe StopIteration exposed sampler cap, not OOM FINDING

ACP probe pt-xt073hk1 measured per-rank peak memory at bs=[1,2,4] → [16.6, 23.9, 38.5] GiB on H100. bs=6 failed with StopIteration — not OOM. Root cause: subject-01 manifest has 20 objects × 5 samples/object, so ObjectGroupedBatchSampler with drop_last=True caps per-object bs at 5. Extrapolated memory ceiling is ~bs=8 (~68 GiB) but we never get there under the old sampler.

Debug container multi-step runs cannot be trusted FAIL

Smoke on debug container produced a valid step 1 (loss=1.373, hand=0.355, svd=0.659) then failed every step 2+ with NVML_SUCCESS == r INTERNAL ASSERT FAILED. ACP-dedicated nodes ran the identical code for 2000 steps cleanly. Saved as feedback memory so future sessions don't chase this as a real bug.
